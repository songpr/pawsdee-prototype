Rule — the header opens the screen. Nothing renders above the section-header slot: no back links, no banners, no trust cards. Recovery and navigation live below the content (a ghost action stacked under the primary CTA) or inline (tappable rows), never above the header. ✅ Do
Header first → content → primary CTA → one ghost action ("แก้ไขข้อมูล · Edit") below it. Tappable summary rows handle inline edits. One edit verb everywhere.
❌ Don't
A "← Back to edit" link above the header, an inline pencil, AND a "Edit details" button — three affordances, two labels. Same action, one name, one or two places.

CTA hierarchy — stack, don't squeeze

One primary CTA per screen, full-width at --text-md (17px). A secondary action (e.g. Skip) stacks full-width below it as a ghost button at the same size — never side-by-side, which clips bilingual labels on narrow screens. Supporting context (expiry, security note) goes in a --text-xs caption under the buttons.

Action grouping — cluster like actions, separate the exits

Some screens carry more than one main action that isn't a simple primary → skip ladder. When two actions serve the same goal — e.g. “don’t miss this booking” via LINE reminders or an Add-to-calendar record — group them as one cluster: a single loud primary, then a same-category secondary as an outlined button directly below it. Keep low-emphasis exits (Skip) and destructive actions (Change / cancel) visually separated as quiet text, so they never compete with the cluster.

Canonical use — the Confirmed screen. “Get reminded” (Link LINE) + “keep a record” (Add to calendar) are one notify-me cluster; Skip and Change/cancel sit apart below it.

Destructive actions — always confirm before committing

Cancelling a booking is destructive — it must never fire on a single tap. The quiet Change / cancel link opens a centred confirm dialog (--shadow-xl, --radius-xl) that restates what’s being cancelled and offers two clearly-weighted choices: keep it (the safe default) and cancel the booking (btn-danger). This is the only place btn-danger appears in the journey. Cancelling frees the slot and returns to booking — reschedule is just cancel + rebook, so no phone call is required.

Booking flow patterns

Used by UX PM

Components specific to the public web-booking & LINE-linking journey (iPhone 13 · 390px · mobile-first). These were previously undocumented and are now standardised so every clinic tenant renders an identical flow.

Date field

Locale overflow wrapper — required. iOS Safari renders date inputs in the device locale: Thai users see "27 มิ.ย. พ.ศ. 2569" (Buddhist Era), which overflows a standard input box. Fix: wrap the <input type="date"> in a position:relative; overflow:hidden container with the border and radius. The input itself gets border:none. The OS renders its locale text naturally; the container clips it cleanly. The native calendar picker still opens on tap. min={todayISO()} prevents past date selection.

Minimum is always the current date — past dates are unselectable.

PDPA disclosure block 2 STANDARD VARIANTS

PDPA distinguishes lawful basis from consent. The rule: Required = inform only (service notice, no toggle). Optional = consent (unchecked checkbox). Only purposes the user can genuinely refuse without losing the service qualify as consent. Confirming writes the actual selected scope to the consents table — never hardcode a purpose's value. Both variants below share one .consent-block + .consent-title scaffold — this is a single standard component, not a one-off box redrawn per screen.

VARIANT A Notice-only — collection and updates
สิทธิ์ข้อมูลของคุณ · Your data rights (PDPA)
PawsDee เก็บชื่อ เบอร์โทร และข้อมูลสัตว์เลี้ยงเพื่อจองและแจ้งเตือนนัดหมายเท่านั้น · เก็บ 24 เดือน · ขอดูหรือลบได้ทุกเมื่อ. We use your name, phone & pet info only to book & send reminders for this visit, kept 24 months. View or delete anytime.

Rule — reappears on update, not just on first collection: PDPA requires the same purpose/retention/rights disclosure whenever collected data is updated or put to a new use, scoped to that use. Two live instances: the Booking form (point of collection — see below) and the SMS backup screen, where an already-collected phone number is repurposed for a new channel (copy: "This number will only be used to send SMS appointment reminders").

Consent vs. lawful basis — the rule
Required ≠ consent.  Optional = consent. If a user cannot refuse a purpose without losing the service, it is not freely given — making it legally invalid as consent (PDPA Art. 19). Use the correct legal basis and the matching UI pattern.

PDPA collection notice — at point of collection (Art. 23)

Personal data (name, phone, pet) is collected on the booking form — so the notice belongs there, before submit, not on the later optional LINE-link step. PDPA Art. 23 requires the purpose, retention period, and the data subject's rights be disclosed at or before collection. The lawful basis is contract performance, so this is a notice, not a consent checkbox — the user does not tick anything and booking is never blocked. This is Variant A of the standard PDPA disclosure block (see Inputs) — the same block reappears, copy scoped to the use, on the SMS confirm screen when an already-collected phone number is repurposed for a new channel.

Confirmation screen pattern

Success uses the compact success-tone section header — icon chip + bilingual title + bilingual description, left-aligned, in the same header slot as every other screen (not a big centred tick that eats the viewport). Then a read-only bilingual summary, a personalised reason-to-act note, and a stacked CTA — the reminder-channel ladder (LINE → SMS backup → Skip) closed by a "booking is safe" caption, no peak-moment countdown.

Add to calendar — the PII-free reminder

"Add to calendar" is rung 3 of the ladder — the record a customer keeps even when they skip LINE and SMS. Two hard rules govern it: (1) the calendar entry carries no personal data, so a forwarded or screenshotted invite leaks nothing; and (2) it is native-first — each platform gets its fewest-tap path, and the options sheet appears only as a desktop / unknown fallback rather than interrupting every user with a chooser.

Why it carries no personal data (PDPA)
A calendar invite is portable by design — users forward it, screenshot it, sync it to shared family calendars. The instant it holds a pet name, owner name, phone, reason, or clinic identity, every one of those becomes an uncontrolled disclosure of personal data (PDPA B.E. 2562, §37 security duty). So we apply data minimisation (§22 — adequate, relevant, limited to necessity): the event is a generic "PawsDee appointment" + time + a 2-hour alert, and nothing else. The actual who/what/where stays private inside LINE.
The exact .ics payload
BEGIN:VEVENT
DTSTART:20260626T030000Z
DTEND:20260626T033000Z
SUMMARY:🐾 เตือนนัดหมาย PawsDee · appointment
DESCRIPTION:ดูรายละเอียดใน LINE · See details in LINE
BEGIN:VALARM
ACTION:DISPLAY
TRIGGER:-PT2H
END:VALARM
END:VEVENT

No ATTENDEE, no LOCATION, no name in SUMMARY/DESCRIPTION. Because nothing personal is processed, this artefact sits outside the consent/lawful-basis matrix entirely — no checkbox, no token, no expiry needed.

Reaching Apple Calendar from inside LINE
iOS only intercepts .ics in Safari — LINE & Chrome run in a WKWebView that silently blocks blob/.ics navigation. The fix: in those in-app browsers the Add-to-calendar action opens a tiny static lander add.html?d=&t=&openExternalBrowser=1 directly (no chooser). LINE reads that flag and re-opens the link in Safari, where the lander rebuilds the same PII-free .ics and the MIME intercept fires. The lander carries only date & time in its URL — still no personal data, so the link is safe to bounce through the OS.
